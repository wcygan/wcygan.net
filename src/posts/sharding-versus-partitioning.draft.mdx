Sharding and partitioning are easy to conflate. PlanetScale's [Sharding vs. partitioning: What's the difference](https://planetscale.com/blog/sharding-vs-partitioning-whats-the-difference) clearly differentiates the two: partitioning splits a table inside one database instance, while sharding spreads data across multiple independent instances.

## Sharding

Sharding spreads one logical dataset across multiple independent database instances, so storage and write load can grow past what one machine can handle. Each shard is a normal database with its own disk and compute, and nothing is shared.

### Routing requests

Because the data lives in different places, something has to decide where each row goes. That decision comes from a _shard key_: the column whose value maps a row to the instance that owns it. A routing layer turns the key into a connection, either inside the application or in a proxy that speaks the database protocol, like [Vitess's VTGate](https://vitess.io/docs/concepts/vtgate/) for MySQL or [Citus's coordinator](https://docs.citusdata.com/en/stable/sharding/data_modeling.html) for Postgres.

The application issues a normal query, and the router sends it to the shard that owns the row.

### Choosing a shard key

The shard key decides how evenly data spreads and which queries can be served by a single shard. [Range sharding](https://www.mongodb.com/docs/manual/core/ranged-sharding/) keeps adjacent keys together, so range scans stay on one shard, but a busy range can overload one instance. [Hash sharding](https://www.mongodb.com/docs/manual/core/hashed-sharding/) scatters keys evenly and avoids hotspots from monotonic keys, but a range query then has to fan out to every shard.

### Hot shards

Even with hashing, skew can concentrate load on one shard. A _hotspot_ is when one key value or one tenant takes most of the traffic, so a single shard saturates while the rest sit idle. A low-cardinality shard key, or one that tracks something like today's date, is the usual cause.

### Crossing shards

The hard part of sharding is any operation that spans shards. A query with the shard key is a targeted lookup; one without it becomes a scatter/gather across every shard. Joins are worse, since related rows often live on different instances, which is why systems like Citus push you to _co-locate_ data on the same key. A transaction across shards needs a [distributed commit](https://en.wikipedia.org/wiki/Two-phase_commit_protocol) to stay atomic, and [resharding](https://vitess.io/docs/concepts/shard/) a live cluster is one of the hardest things to run. [Consistent hashing](https://en.wikipedia.org/wiki/Consistent_hashing) lowers the rebalancing cost by moving only about 1/n of the keys when you add a node.

## Partitioning

Partitioning makes the opposite tradeoff: it stays on one machine and asks the engine, not a router, to do the splitting. It is declarative, so you tell the engine the partition key and method and it routes rows and rewrites queries for you. [PostgreSQL declarative partitioning](https://www.postgresql.org/docs/current/ddl-partitioning.html) and [MySQL partitioning](https://dev.mysql.com/doc/refman/8.0/en/partitioning.html) both offer RANGE for an ordered key like a date, LIST for an explicit set of values, and HASH for even spread when no natural range exists.

```sql
CREATE TABLE events (
  id          bigint,
  created_at  timestamptz NOT NULL
) PARTITION BY RANGE (created_at);
```

### Choosing a partition key

The partition key has to do two jobs that can pull against each other: match the queries you run and spread rows evenly. The Postgres docs suggest the column you most often filter on in a `WHERE` clause. A low-cardinality key (a boolean, a three-value status) or a skewed key (a country column where most rows share one country) leaves you with one giant partition that still gets scanned in full.

### Partition pruning

The payoff for a good key is _partition pruning_. The planner reads each partition's bounds, proves a partition cannot hold any matching rows, and skips it entirely. Pruning follows the partition key, not indexes, so a query has to filter on that key to benefit.

### Lifecycle and limits

Partitioning also makes data lifecycle cheap: dropping a whole partition removes millions of rows almost instantly, because the engine drops a sub-table instead of running a per-row `DELETE`. The limits are that indexes are local to each partition, so a unique constraint has to include the partition key, and that none of this adds a machine. Every partition shares the same CPU, memory, and disk. Partitioning makes one server faster to query and easier to maintain; it is not horizontal scale.

## At LinkedIn

[Espresso](https://engineering.linkedin.com/espresso/introducing-espresso-linkedins-hot-new-distributed-document-store), LinkedIn's distributed document store, is a clean example of the sharded shape. An Espresso database is horizontally partitioned: the leading key in a table's schema is the partition key, and the router hashes it to pick the storage node that serves the request, which is sharding by another name.

Placement is managed by [Apache Helix](https://helix.apache.org/), which spreads partitions evenly across the storage nodes and keeps each partition to one master plus configurable replicas. As the post describes it, Espresso powers around 30 LinkedIn applications and serves millions of records per second at peak. The shard-then-partition composition shows up here too: each storage node could range-partition its big tables by time within the shard.

## Choosing between them

Reach for partitioning when the data fits on one machine and you want pruning on a big table, cheap archival of old rows, or smaller per-partition indexes. Reach for sharding when one machine can no longer hold the data or absorb the writes, and you are willing to pay for cross-shard queries, distributed transactions, and resharding.

The two compose, which is why the line blurs in practice. A shard is just an ordinary database instance, so a large system often shards by tenant or user id across nodes, then range-partitions each shard's big tables by time. You scale across machines with one technique and manage the data within each machine with the other.
