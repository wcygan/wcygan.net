Sharding and partitioning both split one logical dataset into smaller pieces so the database can route work to a narrower slice of data. They are easy to conflate; the simplest distinction is that partitioning splits a table inside one database instance, while sharding spreads data across multiple independent instances.[^planetscale-sharding-partitioning]

## Sharding

Sharding spreads one logical dataset across multiple independent database instances, so storage and write load can grow past what one machine can handle. Each shard is a normal database with its own disk and compute, and nothing is shared.

### Choosing a shard key

The shard key decides how evenly data spreads and which queries can be served by a single shard. Range sharding keeps adjacent keys together, so range scans stay on one shard, but a busy range can overload one instance. Hash sharding scatters keys evenly and avoids hotspots from monotonic keys, but a range query then has to fan out to every shard.

<ShardKeyRangeHashDemo />

### Routing requests

Because the data lives in different places, something has to decide where each row goes. That decision comes from a _shard key_: the column whose value maps a row to the instance that owns it. A routing layer turns the key into a connection, either inside the application or in a proxy that speaks the database protocol, like [Vitess's VTGate](https://vitess.io/docs/concepts/vtgate/) for MySQL or [Citus's coordinator](https://docs.citusdata.com/en/stable/sharding/data_modeling.html) for Postgres.

The application issues a normal query, and the router sends it to the shard that owns the row.

<ShardRequestRouterDemo />

### Hot shards

Even with hashing, skew can concentrate load on one shard. A [_hotspot_](https://docs.pingcap.com/tidb/stable/troubleshoot-hot-spot-issues/) is when one key value or one tenant takes most of the traffic, so a single shard saturates while the rest sit idle. A low-cardinality shard key is the usual cause.

<HotTenantShardDemo />

### Crossing shards

The hard part of sharding is any operation that spans shards. A query without the shard key becomes a scatter/gather across every shard. A transaction across shards needs a [distributed commit](https://en.wikipedia.org/wiki/Two-phase_commit_protocol) to stay atomic, and [resharding](https://vitess.io/docs/concepts/shard/) a live cluster is one of the hardest things to run. [Consistent hashing](https://en.wikipedia.org/wiki/Consistent_hashing) lowers the rebalancing cost by moving only about 1/n of the keys when you add a node.

<ConsistentHashingRingCanvasDemo />

## Partitioning

Partitioning makes the opposite tradeoff: it stays on one machine and asks the database engine to do the splitting. It is declarative, so you tell the engine the partition key and method and it routes rows and rewrites queries for you. [PostgreSQL declarative partitioning](https://www.postgresql.org/docs/current/ddl-partitioning.html), [MySQL partitioning](https://dev.mysql.com/doc/refman/8.0/en/partitioning.html), and [TiDB partitioned tables](https://docs.pingcap.com/tidb/stable/partitioned-table/) all offer RANGE for an ordered key like a date, LIST for an explicit set of values, and HASH for even spread when no natural range exists.

```sql
CREATE TABLE events (
  id          bigint,
  created_at  timestamptz NOT NULL
) PARTITION BY RANGE (created_at);
```

### Choosing a partition key

The partition key has to do two jobs that can pull against each other: match the queries you run and spread rows evenly. The [Postgres docs suggest](https://www.postgresql.org/docs/current/ddl-partitioning.html#DDL-PARTITIONING-DECLARATIVE-BEST-PRACTICES) the column you most often filter on in a `WHERE` clause, and TiDB's [partitioned-table best practices](https://docs.pingcap.com/best-practices/tidb-partitioned-tables-best-practices/) frame the same tradeoff around pruning, write hotspots, and index design. A low-cardinality key (a boolean, a three-value status) or a skewed key (a country column where most rows share one country) leaves you with one giant partition that still gets scanned in full.

<BadPartitionKeyDemo />

### Partition pruning

The payoff for a good key is [_partition pruning_](https://www.postgresql.org/docs/current/ddl-partitioning.html#DDL-PARTITION-PRUNING). The planner reads each partition's bounds, proves a partition cannot hold any matching rows, and skips it entirely. Pruning follows the partition key, not indexes, so a query has to filter on that key to benefit.

## Conclusion

Sharding and partitioning both route rows by key. A partition key groups rows inside one database so the engine can route writes and skip irrelevant chunks. A shard key chooses which independent storage node owns the row or request.[^row-routing]

| Pattern                | Key                | Reason                                                                                                                                                    |
| ---------------------- | ------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Merchant SaaS          | `shop_id`          | In Shopify's shop-owned tables, `shop_id` identifies the shop; shard balancing moves rows for that shop to another shard.[^shopify-shop-id]               |
| Multi-tenant SaaS      | `tenant_id`        | The Citus tenant example distributes tenant tables by `tenant_id`, keeping same-tenant joins on one worker.[^citus-tenant-id]                             |
| DM search              | `user_id`          | Discord reindexed direct messages by `user_id`, so cross-DM search can target one user's messages instead of every conversation.[^discord-message-search] |
| Warehouse partitioning | `transaction_date` | BigQuery can prune partitions when a query filters the `transaction_date` partition key, so date filters scan less data.[^bigquery-partition-pruning]     |

Choose keys that appear in filters or joins, have high cardinality, avoid skew, and colocate rows queried together. Use time keys for pruning or retention; use entity keys when distributing time-series data. Avoid low-cardinality distribution keys such as a three-value `status` column.[^key-checklist]
