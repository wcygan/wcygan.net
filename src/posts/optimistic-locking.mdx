[Optimistic Locking](https://en.wikipedia.org/wiki/Optimistic_concurrency_control) prevents a stale write from silently overwriting newer state
by storing a version beside the row, reading that version with the data, and
requiring the version to still match when the write commits. If the update
affects one row, the worker saved fresh state; if it affects zero rows, another
worker changed the row first and the application should reread the latest row
before retrying.

<OptimisticLockingRaceDemo />

The `UPDATE` SQL carries the row version you read into the `WHERE` clause.
This makes the `UPDATE` behave like a [compare-and-swap](https://en.wikipedia.org/wiki/Compare-and-swap) on one row:

```sql
UPDATE inventory_items
SET
  available = available - 1,
  version = version + 1
WHERE sku = 'SKU-42'
  AND available > 0
  AND version = :version;

SELECT ROW_COUNT();
```

In this example, <code>ROW_COUNT() = 0</code> means the version guard missed.
Because <code>available &gt; 0</code> is also part of the guard, production code
should reread the row before deciding whether the failed update was a stale
version, no remaining stock, or a missing SKU.

It is "optimistic" because each worker proceeds as if the row will stay
unchanged, then uses the version check to detect when that bet was wrong.

With pessimistic row locking (`SELECT ... FOR UPDATE`), Worker A would lock the inventory row before
thinking about the reservation. Worker B would wait for A to commit or roll
back, then read the newer `available` and `version` values before deciding
whether to reserve anything. That avoids the stale-write retry path, but it
keeps a database lock open while application work happens.

```sql
-- Worker A read v7.
-- available 2.
UPDATE inventory_items
SET
  available = available - 1,
  version = version + 1
WHERE sku = 'SKU-42'
  AND available > 0
  AND version = 7;

SELECT ROW_COUNT();
-- 1 row

-- Worker B also read v7.
-- Its stale write is rejected.
UPDATE inventory_items
SET
  available = available - 1,
  version = version + 1
WHERE sku = 'SKU-42'
  AND available > 0
  AND version = 7;

SELECT ROW_COUNT();
-- 0 rows

-- Worker B rereads v8.
-- It recomputes and retries.
UPDATE inventory_items
SET
  available = available - 1,
  version = version + 1
WHERE sku = 'SKU-42'
  AND available > 0
  AND version = 8;

SELECT ROW_COUNT();
-- 1 row
```

## At LinkedIn

We use [Temporal Workflows](/durable-execution#at-linkedin) to orchestrate order placement. Some of these steps may execute in parallel and update state in the order database. This implies that there is a potential for a [data race](https://en.wikipedia.org/wiki/Race_condition), where, with optimistic locking, one writer will win, and others will lose. The combination of Temporal Retries and Optimistic Locking allow for workflows to proceed even if failures are encountered.
