The N+1 SQL query problem shows up when application code asks for a list of
records, then loops through that list and runs another query for each row,
performing 1 + N database round trips.

The repeated trips, query execution, and result transfers usually cost more than one larger query the database can optimize. The issue can be resolved by using a `JOIN` to fetch the related rows in one pass before
shaping them into the structure the application wants.[^planetscale-n-plus-one]

At a high level, the N+1 version moves the loop across the database boundary:

```python
def n_plus_one(db):
    users = db.query("SELECT id, name FROM users LIMIT 10")

    rows = []
    for user in users:
        orders = db.query(
            "SELECT id, total FROM orders WHERE user_id = ?",
            [user["id"]],
        )
        rows.append({**user, "orders": orders})

    return rows
```

The batched version keeps the same page shape, but asks for all related rows in
one child query:

```python
def batch_query(db):
    users = db.query("SELECT id, name FROM users LIMIT 10")
    user_ids = [user["id"] for user in users]

    orders = db.query(
        "SELECT id, user_id, total FROM orders WHERE user_id IN (?)",
        [user_ids],
    )

    return attach_orders(users, orders)
```
