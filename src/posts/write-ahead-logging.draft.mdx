[The Log: What every software engineer should know about real-time data's unifying abstraction](https://engineering.linkedin.com/distributed-systems/log-what-every-software-engineer-should-know-about-real-time-datas-unifying) explains that logs provide two properties databases and distributed systems both rely on: a durable record of what happened and an ordered sequence for replay.

<WalWritePathDemo />

After a crash, recovery reads durable WAL records and rebuilds the missing in-memory state.

<WalReplayDemo />

Two edge cases make the rule practical:

- If both the log record and page reached disk, redo can skip the update because the page already carries `pageLSN = 110`.
- If a crash tears a log record, recovery stops at the last complete record. Length fields and checksums make that boundary detectable.

## Why this matters

WAL makes commits cheaper because the database can flush a sequential log record instead of forcing every modified page to disk immediately. The tradeoff is restart work: recovery scans enough log to redo changes that made it to the WAL but not to the final data pages.
